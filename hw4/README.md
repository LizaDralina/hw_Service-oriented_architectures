Централизованная Notification Platform для онлайн-банка


## 1. Функциональные требования

### FR-1. Централизованная отправка уведомлений
Все продуктовые команды банка должны отправлять уведомления пользователям через единую Notification Platform, а не напрямую через свои каналы.

**Зачем бизнесу:**
- уменьшает дубли и задержки
- дает централизованный контроль
- снижает количество жалоб

### FR-2. Поддержка разных типов уведомлений
Платформа должна уметь обрабатывать три типа уведомлений:
- транзакционные
- сервисные
- маркетинговые

Для каждого типа должны применяться свои правила приоритета, доставки и пользовательских настроек.

**Зачем бизнесу:**
- критические сообщения не должны конкурировать с рекламными
- позволяет повысить качество коммуникации и retention

### FR-3. Поддержка нескольких каналов доставки
Платформа должна уметь доставлять уведомления через:
- push
- SMS
- email

Для одного уведомления система должна выбирать подходящий канал с учетом важности сообщения и доступности контактов пользователя.

**Зачем бизнесу:**
- повышает шанс доставки
- дает гибкость и снижает стоимость, если можно использовать более дешевый канал

### FR-4. Автоматическое переключение на резервный канал для критичных уведомлений
Если критичное уведомление не доставлено через основной канал, система должна автоматически попытаться доставить его через резервный канал.

**Зачем бизнесу:**
- обеспечивает гарантированную доставку критичных уведомлений
- снижает риск пропуска важных сообщений о переводах и списаниях

### FR-5. Учет пользовательских настроек
Пользователь должен иметь возможность управлять уведомлениями:
- маркетинговые — можно отключить полностью
- сервисные — можно отключать по каналам
- транзакционные критические — нельзя отключить полностью, но можно выбрать предпочтительный канал, если при этом остается хотя бы один обязательный канал доставки

правило: для критичных уведомлений SMS отключить нельзя, так как это обязательный резервный канал.

**Зачем бизнесу:**
- уменьшает раздражение пользователей
- снижает количество жалоб
- при этом сохраняет безопасность

### FR-6. Предотвращение дублей
Если одно и то же бизнес-событие пришло в платформу повторно, пользователь не должен получать одинаковое уведомление несколько раз.

**Зачем бизнесу:**
- снижает жалобы
- уменьшает недоверие к банку

### FR-7. Отслеживание статусов доставки
Инициирующая система, служба поддержки и операторы должны видеть статус уведомления:
- принято
- отправляется
- доставлено
- не доставлено
- доставлено через резервный канал
- исчерпаны попытки

**Зачем бизнесу:**
- упрощает поддержку
- повышает прозрачность
- помогает разбирать инциденты

### FR-8. Массовая рассылка
Платформа должна поддерживать массовые кампании до 1 млн пользователей с возможностью планирования отправки и контроля нагрузки.

**Зачем бизнесу:**
- поддержка акций и маркетинга
- рост retention


## 2. Нефункциональные требования

### NFR-1. Задержка для критичных уведомлений
Для транзакционных уведомлений:
- **p95** время от приема запроса платформой до старта первой попытки доставки: **не более 1 секунды**
- **p99** время до достижения финального статуса "доставлено хотя бы по одному каналу" или "исчерпаны попытки": **не более 30 секунд**

### NFR-2. Доступность
Для контура приема и оркестрации критичных уведомлений месячная доступность должна быть **не ниже 99.95%**.

### NFR-3. Надежность и отсутствие потерь
Для критичных уведомлений:
- потеря принятых платформой уведомлений — **не более 1 на 10 000 000**
- **RPO = 0**
- **RTO ≤ 15 минут**

### NFR-4. Масштабируемость
Система должна выдерживать:
- до **5 000** входящих запросов в секунду по платформе в целом
- до **2 000** запросов в секунду по критичным уведомлениям
- до **4 000** попыток доставки в секунду для критичных уведомлений в сценарии массового failover
- маркетинговую кампанию **1 млн пользователей за 15 минут**

### NFR-5. Защита от дублей
Для критичных уведомлений доля дублей, дошедших до пользователя, должна быть **менее 0.1%**.

### NFR-6. Наблюдаемость
Для 100% критичных уведомлений должна быть доступна сквозная трассировка по notification_id:
- история попыток доставки
- выбранные каналы
- причины failover
- финальный статус

Обновление метрик и статусов в мониторинге — **не позже 1 минуты**.

### NFR-7. Безопасность данных
- 100% передачи данных — только по защищенным каналам
- 100% чувствительных данных пользователя — в зашифрованном виде в хранилищах
- доступ к данным уведомлений — только по ролевой модели


## 3. Архитектурно значимые требования (ASR)

### ASR-1. Низкая задержка доставки критичных уведомлений
**Связанные требования:**
- FR-4 (автоматический failover)
- NFR-1 (p95 старт первой попытки ≤ 1 сек)

**Почему влияет на архитектуру:** Требует выделенного "горячего" контура обработки, минимального числа синхронных операций, приоритизации критичных уведомлений и отказа от тяжелой пакетной обработки в общем потоке.

### ASR-2. Гарантированная доставка критичных уведомлений с failover и защитой от дублей
**Связанные требования:**
- FR-4 (failover)
- FR-6 (защита от дублей)
- FR-7 (отслеживание статусов)
- NFR-3 (надежность)
- NFR-5 (доля дублей < 0.1%)

**Почему влияет на архитектуру:** Нужны долговременное хранение состояния доставки, механизм оркестрации попыток, таймауты, повторные попытки, идемпотентность и обработка подтверждений от внешних провайдеров.

### ASR-3. Масштабируемость и изоляция нагрузок
**Связанные требования:**
- FR-8 (массовая рассылка)
- NFR-4 (масштабируемость)

**Почему влияет на архитектуру:** Требует разделения потоков по приоритетам, горизонтального масштабирования и независимых контуров обработки, чтобы маркетинг не ломал транзакционные уведомления.

### ASR-4. Сквозная наблюдаемость и аудит
**Связанные требования:**
- FR-7 (статусы)
- NFR-6 (наблюдаемость)
- NFR-7 (безопасность данных)

**Почему влияет на архитектуру:** Нужны единые идентификаторы, журнал событий доставки, трассировка, централизованные метрики и возможность аудита по каждому уведомлению.


## 4. Ключевые архитектурные вопросы

### AQ-1. Как реализовать логику failover: через централизованный оркестратор или через event-driven choreography?
- **Порождается:** ASR-1, ASR-2, ASR-4
- **Почему важно:** Этот выбор определяет, где хранится состояние доставки, как реализуются таймауты, насколько просто доказывать корректность процесса и расследовать инциденты.

### AQ-2. Что считать "доставкой" для каждого канала?
- **Порождается:** ASR-2, ASR-4
- **Почему важно:** Для SMS можно получить delivery receipt, для push — часто только подтверждение провайдера или ACK от мобильного приложения, для email — только acceptance/bounce. От определения "успеха" зависит логика failover и обещания бизнесу.

### AQ-3. Как изолировать критические уведомления от маркетинговых кампаний?
- **Порождается:** ASR-1, ASR-3
- **Почему важно:** Если все типы уведомлений пойдут через общую очередь и общий пул воркеров, массовая рассылка может сорвать SLA для переводов и списаний.

### AQ-4. Как исключить дубли при повторной отправке события, ретраях и поздних callback'ах от провайдеров?
- **Порождается:** ASR-2, ASR-4
- **Почему важно:** Без единой модели идемпотентности пользователь может получить несколько одинаковых сообщений, особенно в сценариях failover.


## 5. Архитектурные последствия ASR

### Для ASR-1. Низкая задержка
**Последствия:**
- асинхронная обработка после быстрого приема запроса
- отдельный приоритетный контур для критичных уведомлений
- минимизация синхронных зависимостей на горячем пути
- кэширование пользовательских настроек и контактных данных
- выделенные очереди/воркеры для critical traffic

### Для ASR-2. Гарантированная доставка и failover
**Последствия:**
- хранение состояния доставки как явной state machine / workflow
- идемпотентность на входе и на каждом канале
- таймауты и повторные попытки
- получение callback/receipt от провайдеров
- механизм подавления дублей при поздних ответах

### Для ASR-3. Масштабируемость и изоляция
**Последствия:**
- раздельные очереди/топики по типам уведомлений
- независимое масштабирование воркеров
- rate limit и quota по каналам и провайдерам
- backpressure
- возможно отдельные контуры для bulk и critical

### Для ASR-4. Наблюдаемость и аудит
**Последствия:**
- единый notification_id, attempt_id, correlation_id
- журнал всех значимых событий доставки
- централизованные метрики и алерты
- distributed tracing
- аудит изменений пользовательских настроек


## 6. Архитектурные решения, которые не подходят

### Неподходящее решение 1. Прямые синхронные вызовы провайдеров из продуктовых сервисов
**Нарушаемый ASR:** ASR-2, ASR-4, частично ASR-1

**Почему не подходит:**
- нет централизованного контроля
- сложно реализовать failover
- нет единой защиты от дублей
- высокий риск потери сообщений при сбоях
- невозможно получить единый аудит и статусы

### Неподходящее решение 2. Одна общая очередь и один общий пул воркеров для всех типов уведомлений
**Нарушаемый ASR:** ASR-1, ASR-3

**Почему не подходит:**
- маркетинговая рассылка может забить очередь
- критические уведомления потеряют приоритет
- невозможно гарантировать SLA на транзакционные сообщения

### Неподходящее решение 3. Использование только push для критичных уведомлений
**Нарушаемый ASR:** ASR-2

**Почему не подходит:**
- push зависит от токена устройства, мобильной ОС, состояния приложения и доступности провайдера
- нет надежной гарантии доставки в условиях нестабильности
- отсутствует надежный резервный канал


## 7. Неопределенности и архитектурные риски

### Риск 1. Неясно, насколько надежны delivery receipt и callback'и у SMS/email-провайдеров

**Что неизвестно:**
- точность статусов delivered
- задержки callback'ов
- поведение при частичных сбоях

**Как проверить:**
- провести пилот с двумя SMS-провайдерами и одним email-провайдером
- собрать метрики по delivered rate, callback lag, error rate
- провести load testing

### Риск 2. Неясно, какое определение "доставлено" бизнес готов считать приемлемым для push/email

**Что неизвестно:**
- достаточно ли acceptance от провайдера
- нужен ли ACK от мобильного приложения
- можно ли считать email резервным гарантированным каналом

**Как проверить:**
- согласовать policy с бизнесом, безопасностью и support
- провести PoC с mobile ACK для push
- зафиксировать SLA по каждому каналу

### Риск 3. Неизвестны реальные пиковые нагрузки в дни аномальной активности

**Что неизвестно:**
- пиковая кратность всплеска относительно среднего дня
- корреляция с маркетинговыми кампаниями
- объем failover при падении основного провайдера

**Как проверить:**
- взять исторические банковские пики
- провести нагрузочное тестирование на 2x–3x от прогнозируемого пика
- сделать game day со сценарием недоступности push-провайдера

## 1. Контекст

### Проблема

В текущей архитектуре разные команды банка отправляют уведомления самостоятельно (напрямую через провайдеров). Это приводит к:

- **Задержкам** — каждая команда реализует свою логику, нет единого SLA
- **Дублям** — одно событие может отправиться несколько раз разными сервисами
- **Отсутствию контроля** — невозможно гарантировать доставку критичных уведомлений
- **Высокой стоимости** — все используют SMS как основной канал, хотя push дешевле в 10-20 раз

### Почему важно сейчас

- Рост числа транзакций в онлайн-банке (+40% год к году)
- Регуляторные требования к подтверждению операций
- Жалобы пользователей на отсутствие уведомлений о списаниях

### Кто затронут

- **Пользователи** — получают критические уведомления (переводы, списания)
- **Продуктовые команды** — интегрируются через единую платформу
- **Служба поддержки** — видит статусы доставки
- **Маркетинг** — не влияет на критичный трафик

## 4. Варианты решения

### Вариант 1: Централизованный workflow-оркестратор (Temporal)

#### Описание архитектуры

Каждое критичное уведомление обрабатывается как отдельный **долговечный workflow** в Temporal.

**Компоненты:**
- **API Gateway** — прием запросов, генерация notification_id
- **Kafka** — буферизация, отдельный топик для критичных уведомлений
- **Temporal Cluster** — оркестрация workflow, таймауты, retry
- **Temporal Workers** — отправка push, ожидание ACK, failover на SMS
- **Redis** — кэш пользовательских настроек
- **PostgreSQL** — аудит и история попыток
- **Cassandra** — хранение состояния workflow

**Основной сценарий:**
1. Продуктовый сервис отправляет запрос -> API генерирует notification_id и стартует workflow
2. Workflow получает из Redis настройки пользователя, определяет primary=push, fallback=SMS
3. Workflow отправляет push, устанавливает таймаут 5 секунд
4. Приходит delivery receipt -> статус DELIVERED, запись в аудит

**Failover сценарий:**
1. За 5 секунд подтверждение не пришло -> workflow переключается на SMS
2. Отправка SMS, ожидание receipt (2-10 секунд)
3. Статус DELIVERED_VIA_FALLBACK

**Защита от дублей:** Проверка business_event_id в Temporal. При повторном запросе возвращается существующий notification_id.

#### Как решение удовлетворяет ASR

| ASR | Как выполняет |
|-----|---------------|
| ASR-A | Асинхронный старт, кэш в Redis, p95 старта ~50 мс |
| ASR-B | Workflow ждет receipt, таймаут -> failover, idempotency key |
| ASR-C | Отдельный Kafka topic + отдельный пул workers |
| ASR-D | Temporal Web UI + Jaeger + PostgreSQL |

#### Trade-offs варианта 1

| Аспект | Плюсы (+) | Минусы (-) |
|--------|-----------|------------|
| Сложность реализации | Таймауты и retry встроены | Нужно изучить Temporal, новая инфраструктура |
| Наблюдаемость | Temporal UI из коробки | — |
| Долговечность | Workflow переживает рестарты | Cassandra требует операционной нагрузки |
| Задержка | p95 < 100 мс, нет rebalance | При RPS >10k нужен тюнинг |
| Операционные расходы | — | Новый стек (Temporal + Cassandra) |

### Вариант 2: Event-driven state machine на Kafka

#### Описание архитектуры

Состояние каждого уведомления хранится в **Kafka Streams state store** (RocksDB), переходы между состояниями — через обработку событий из разных топиков.

**Компоненты:**
- **API Gateway** — прием запросов
- **Kafka Topics:** critical-ingestion, critical-pending, critical-callbacks
- **Kafka Streams** — stateful обработка, таймауты, дедупликация
- **Push/SMS Workers** — отправка уведомлений провайдерам
- **Redis** — кэш user preferences
- **ClickHouse** — аудит

**Основной сценарий:**
1. API отправляет событие в critical-ingestion, возвращает 202 Accepted
2. Kafka Streams проверяет дубли в state store, обогащает данными из Redis
3. Push Worker отправляет push, производит событие в critical-callbacks

**Failover сценарий:**
1. Таймаут реализуется через оконную операцию: windowedBy(TimeWindows.of(Duration.ofSeconds(5)))
2. При отсутствии receipt генерируется событие "timeout" в critical-pending с флагом fallback
3. SMS Worker отправляет SMS

#### Как решение удовлетворяет ASR

| ASR | Как выполняет |
|-----|---------------|
| ASR-A | В обычном режиме <100 мс, но риск при rebalance |
| ASR-B | State store для dedup, оконные операции для таймаутов |
| ASR-C | Отдельные топики для critical vs marketing |
| ASR-D | JMX-метрики Kafka, Prometheus, нет единого UI |

#### Trade-offs варианта 2

| Аспект | Плюсы (+) | Минусы (-) |
|--------|-----------|------------|
| Сложность реализации | Если Kafka уже есть — нет новой инфраструктуры | Таймауты на Kafka Streams сложны |
| Наблюдаемость | Стандартные инструменты | Нет единого UI для одного уведомления |
| Долговечность | Kafka хранит события | State store (RocksDB) может потеряться |
| Задержка | В обычном режиме отлично | **Риск:** при rebalance задержка до 5-10 сек |
| Стоимость разработки | Нет новой инфраструктуры | Код таймаутов пишется вручную |

## 6. Выводы

### Рекомендация

**Выбран вариант 1: Централизованный workflow-оркестратор на Temporal**

### Обоснование выбора

1. **ASR-A (низкая задержка)** — Temporal гарантирует быстрый старт workflow без rebalance, нет риска внезапного роста задержки.

2. **ASR-B (failover и dedup)** — встроенные таймауты и retry идеально подходят для сценария "push -> таймаут 5 сек -> SMS".

3. **ASR-D (наблюдаемость)** — Temporal Web UI показывает каждый workflow, историю, причины failover. Для поддержки это критично.

4. **Долговечность** — workflow восстанавливаются из Cassandra после падения воркеров.

### Почему не выбран вариант 2 (Kafka)

- Реализация таймаутов на Kafka Streams сложна и хрупка
- Нет единого UI для наблюдаемости
- Риск нарушения NFR1 при rebalance consumer-группы
- State store (RocksDB) может потеряться при падении воркера

## 8. Приложения

### Глоссарий

| Термин | Определение |
|--------|-------------|
| **ACK** | Acknowledgment — подтверждение получения уведомления |
| **ASR** | Architecturally Significant Requirement |
| **Failover** | Переключение на резервный канал при отказе основного |
| **p95 / p99** | 95-й и 99-й перцентили |
| **SLA** | Service Level Agreement |
| **Temporal** | Open-source workflow orchestration engine |

### Принятые бизнес-правила

1. Критичные транзакционные уведомления нельзя отключить полностью.
2. Для каждого клиента банка существует verified phone number.
3. SMS не может быть отключен как резервный канал для критичных.
4. По умолчанию: primary = push, fallback = SMS.
5. Для push успех = mobile ACK или delivery receipt от провайдера.
6. Для SMS успех = delivery receipt = delivered.
7. Доля дублей у пользователя < 0.1%.
