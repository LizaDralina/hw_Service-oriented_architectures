## 1) Домены и зона ответственности 

### 1.1 Users (Пользователи)
**Ответственность:**
- учетные записи пользователей (покупатель/продавец)
- профили (имя, контакты и т.п.)

### 1.2 Catalog (Каталог)
**Ответственность:**
- товары, категории, атрибуты
- цены как часть карточки товара (на уровне данных)
- операции продавца по управлению товарами 

### 1.3 Feed / Personalization (Лента и персонализация)
**Ответственность:**
- выдача ленты товаров пользователю
- персонализация (упрощенно): популярное/новинки + сигналы поведения (просмотры, покупки)

### 1.4 Orders (Заказы)
**Ответственность:**
- создание заказа
- хранение состава заказа (позиции)
- жизненный цикл заказа (статусы)

### 1.5 Payments (Платежи)
**Ответственность:**
- учет платежей, транзакций, статусов
- взаимодействие с внешним платежным провайдером (эквайринг)

### 1.6 Notifications (Уведомления)
**Ответственность:**
- отправка уведомлений о событиях (статусы заказа/платежа)

## 2) Сервисы и распределение доменов

### 2.1 Карта “домен → сервис”
- **Users** → **User Service**
- **Catalog** → **Catalog Service**
- **Feed/Personalization** → **Feed Service**
- **Orders** → **Order Service**
- **Payments** → **Payment Service**
- **Notifications** → **Notification Service**

Дополнительные технические контейнеры:
- **API Gateway** — единая точка входа для клиентов
- **Message Broker** (Kafka/RabbitMQ) — асинхронные события между сервисами

### 2.2 Логика такого разбиения
Разбиение выполнено по принципам:
1) **Bounded Context**: домены слабосвязаны и имеют разную скорость изменений (каталог, лента, заказы, платежи, уведомления) — удобнее развивать отдельно.
2) **Изоляция данных**: каждый сервис — единственный владелец своих данных (нет shared DB), что снижает риск конфликтов и упрощает эволюцию схем.
3) **Событийные интеграции там, где это естественно**: уведомления о статусах заказа/платежа удобнее делать через async события (не блокируя основной поток заказа).
4) **API Gateway** разгружает клиентов: единая точка входа, маршрутизация, в будущем — агрегация/авторизация (без реализации на данном этапе).

## 3) Границы владения данными 

Принцип: **каждый сервис владеет собственной БД/хранилищем**. Другие сервисы не читают/не пишут напрямую в чужие базы.

| Сервис | Чем владеет (данные) | За что отвечает |
|---|---|---|
| API Gateway (BFF) | - | входная точка, маршрутизация запросов к сервисам |
| User Service | пользователи, профили | управление пользователями и их профилями |
| Catalog Service | товары, категории, атрибуты, цены | управление каталогом и карточками товаров |
| Feed Service | настройки/профиль персонализации | формирование и выдача ленты |
| Order Service | заказы, позиции заказа, статусы | создание заказа и изменение статусов |
| Payment Service | платежи, транзакции, статусы | учет/проведение платежей, связь с провайдером |
| Notification Service | - | доставка уведомлений пользователям |
| Message Broker | очереди | доставка событий между сервисами |

у каждого доменного сервиса своя БД.

## 4) Взаимодействия сервисов (sync/async)

### 4.1 Синхронные взаимодействия (HTTP)
Клиенты (Web/Mobile) работают через **API Gateway**:

- Клиент → **API Gateway** (HTTPS)
- **API Gateway** → **User Service** (HTTP): профиль
- **API Gateway** → **Catalog Service** (HTTP): чтение/управление товарами
- **API Gateway** → **Feed Service** (HTTP): получить ленту
- **API Gateway** → **Order Service** (HTTP): создать/получить заказ
- **API Gateway** → **Payment Service** (HTTP): инициировать платеж/получить статус


### 4.2 Асинхронные взаимодействия (Message Broker)
События домена публикуются в **Message Broker**:

**Публикация:**
- **Order Service** → Broker: **OrderCreated**, **OrderStatusChanged**
- **Payment Service** → Broker: **PaymentSucceeded**, **PaymentFailed**


**Подписки:**
- Broker → **Notification Service**: события заказов/платежей → отправка уведомлений
